# 字幕组任务管理 Bot 设计文档

## 一、项目背景

### 1.1 团队现状

字幕组日常工作以"集"为单位推进，每集需经过多道工序协作完成。目前任务分配和进度跟踪依赖人工在腾讯文档智能表中手动填写，存在信息滞后、漏接、重复接活等问题。

### 1.2 技术选型

| 组件 | 选型 | 说明 |
|------|------|------|
| 数据存储 | 腾讯文档智能表 | 每个番剧一个子表，表名格式：`番剧名_工作表` |
| 群聊平台 | QQ 群 | 每个番剧绑定一个工作群，另设总群查看全局进度 |
| Bot 框架 | NoneBot2 + NapCat | NapCat 提供 QQ 协议适配，NoneBot2 处理命令逻辑 |
| 文档 API | 腾讯文档开放平台 Open API | 个人开发者可用，OAuth2.0 授权，每日 2000 次调用额度 |

### 1.3 API 额度约束

腾讯文档开放平台个人开发者每日限额 **2000 次 API 调用**。设计中需注意：

- Bot 启动时拉取一次全量数据缓存至内存，后续查询优先读缓存
- 写操作（接活、完成等）实时调用 API，同时同步更新本地缓存
- 查询类命令（`/进度`、`/我的任务`、`/待接`）全部走本地缓存，不消耗 API 额度
- 设置定时全量同步（如每 30 分钟一次），防止缓存与远端漂移
- 预估日常消耗：新建集 ×1~2 次（每次约 10~20 条写入）+ 接活/完成 ×20~30 次 + 定时同步 ×48 次 ≈ 200~300 次/天，远低于 2000 上限

### 1.4 设计原则：模块化与可迁移

本插件未来可能迁移至飞书多维表格、或从 NoneBot2 迁移至 AstrBot 等其他框架。因此架构上严格分层，各层通过接口（抽象类）解耦：

```
┌─────────────────────────────────────────────┐
│             commands（命令解析层）             │ ← 依赖 Bot 框架（NoneBot2 / AstrBot）
├─────────────────────────────────────────────┤
│           task_manager（业务逻辑层）           │ ← 纯业务，不依赖任何框架或存储实现
├─────────────────────────────────────────────┤
│            storage（存储抽象层）               │ ← 抽象接口
│  ┌──────────────┐  ┌──────────────────────┐ │
│  │ TencentDoc   │  │ FeishuSheet（未来）   │ │ ← 具体实现，可替换
│  └──────────────┘  └──────────────────────┘ │
├─────────────────────────────────────────────┤
│           messenger（通知抽象层）              │ ← 抽象接口
│  ┌──────────────┐  ┌──────────────────────┐ │
│  │ QQMessenger  │  │ FeishuMessenger      │ │ ← 具体实现，可替换
│  └──────────────┘  └──────────────────────┘ │
└─────────────────────────────────────────────┘
```

迁移时只需替换最底层的实现，业务逻辑层和命令层不受影响。

## 二、智能表结构设计

### 2.1 子表规则

每个番剧对应一个子表，命名格式：`{番剧名}_工作表`

示例：`淡岛百景_工作表`、`孤独摇滚_工作表`

### 2.2 番剧与 QQ 群绑定

#### 群类型

| 群类型 | 说明 | 命令权限 | 通知推送 |
|--------|------|----------|----------|
| **工作群** | 每个番剧绑定的专属群 | 可执行所有命令（接活/完成/新建等） | 接收该番剧的所有通知（任务创建、工序推进等） |
| **总群** | 全局配置一个，所有番剧共享 | 可执行只读查询命令（`/进度`、`/我的任务`、`/待接` 查看所有番剧） | **不接收任何推送通知** |

设计要点：
- 总群通过配置项 `main_group_id` 指定，不占用绑定关系
- 在总群发送 `/进度 淡岛百景 07` 可查看任何番剧，无需绑定
- 在总群发送 `/接活` 等写操作命令会被拒绝，提示"请到对应工作群操作"
- 一个工作群可以绑定多个番剧，但一个番剧只能绑定一个工作群

#### 绑定方式

支持两种绑定方式，适配不同场景：

**方式一：按番剧名绑定**（智能表子表名符合 `{番剧名}_工作表` 命名规范时）

```
/绑定 淡岛百景
```

Bot 自动查找腾讯文档中名为 `淡岛百景_工作表` 的子表并关联。

**方式二：按 ID 绑定**（子表命名不规范、或需要指定特定文档时）

```
/绑定id <fileID> <sheetID> [别名]
```

直接通过腾讯文档的 fileID 和 sheetID 绑定，可选设置一个别名用于后续命令中引用：

```
/绑定id abc123 sheet456 淡岛百景
```

后续命令中使用别名即可：`/接活 淡岛百景 07 翻译 P1`

#### 绑定数据结构

```json
{
  "main_group_id": 111111111,
  "bindings": {
    "淡岛百景": {
      "group_id": 123456789,
      "file_id": "abc123",
      "sheet_id": "smartsheet_xxx",
      "bound_by": 987654321,
      "bound_at": "2026-05-25T14:00:00"
    },
    "孤独摇滚": {
      "group_id": 223456789,
      "file_id": "abc123",
      "sheet_id": "smartsheet_yyy",
      "bound_by": 987654321,
      "bound_at": "2026-05-20T10:00:00"
    }
  }
}
```

### 2.3 表头字段

| 字段名 | 类型 | 说明 | 示例值 |
|--------|------|------|--------|
| 类型 | 单选 | 工序类型 | 翻译 / 时轴 / 校对 / 后期 / 监制 / 压制 |
| 集数 | 文本 | 集数编号，从项目名中独立出来便于查询过滤 | `07`、`OVA1`、`OP`、`ED` |
| 分段 | 文本 | 数字串：分段工序填 `1`/`2`/`3`...；不分段的工序填 `0`（D12） | `1`、`2`、`0` |
| 组员 | 文本 | 接活人标识（Bot 写入 QQ 号；手动填表可能是昵称） | `123456789` 或 `小明` |
| 进度 | 单选 | 当前任务状态 | 未分配 / 已分配 / 进行中 / 已完成 / 归档 |
| 完成时间 | 日期 | 标记完成时自动写入 | `2026-05-25` |
| 备注 | 文本 | 自由备注 | `加急`、`需要配合第6集` |

### 2.4 组员字段与通知的兼容策略

由于需要兼容手动填表，组员字段的值可能是 QQ 号，也可能是昵称或其他文本。Bot 在发送通知时按以下逻辑处理：

```
读取组员字段值
  │
  ├─ 纯数字且能匹配到群成员 QQ 号 → @该成员（CQ at）
  │
  └─ 非纯数字 / 数字但不在群内 → 作为文本显示原值
```

示例：
- 组员字段为 `123456789`（群内成员） → `@小明 完成了 翻译 P1 ✅`
- 组员字段为 `小明`（手动填的昵称） → `小明 完成了 翻译 P1 ✅`
- 下游通知同理：`校对现在可以接了！` 如果校对员已接活且组员是 QQ 号，则 @；否则发文本

### 2.5 为什么拆"集数"和"分段"

原始设计中"项目"字段承载了番剧名+集数+分段信息（如 `淡岛百景07（0-8）`），格式不统一，Bot 解析困难。拆分后：

- **番剧名**已由子表名承载，无需重复
- **集数**独立为字段，支持 API 按集数精确过滤
- **分段**独立为字段，便于判断"该工序是否需要分段""某集的所有分段是否完成"

## 三、工序流水线（可自定义）

### 3.1 设计思路

不同番剧的工序流程可能不同：有的翻译分段时轴不分段，有的没有后期环节，有的校对之前只需要翻译不需要时轴。因此工序流水线不硬编码，而是**按番剧配置**。

每个番剧的流水线配置包含：

```json
{
  "淡岛百景": {
    "pipeline": [
      {
        "stage": "翻译",
        "segment": true,
        "depends_on": []
      },
      {
        "stage": "时轴",
        "segment": true,
        "depends_on": []
      },
      {
        "stage": "校对",
        "segment": false,
        "depends_on": ["翻译", "时轴"]
      },
      {
        "stage": "后期",
        "segment": false,
        "depends_on": ["校对"]
      },
      {
        "stage": "监制",
        "segment": false,
        "depends_on": ["后期"]
      },
      {
        "stage": "压制",
        "segment": false,
        "depends_on": ["监制"]
      }
    ]
  }
}
```

每个工序节点有三个属性：
- `stage`：工序名称
- `segment`：是否按时间段分段（true = 新建集时按 1/2/3/... 展开，false = 只生成一条「分段=0」记录）（D12）
- `depends_on`：前置工序列表（空 = 无依赖，可立即接活）

### 3.2 典型流水线模板

**标准流程**（翻译+时轴均分段）：

```
翻译（分段） ──┐
               ├──→ 校对 → 后期 → 监制 → 压制
时轴（分段） ──┘
```

**简化流程**（无后期）：

```
翻译（分段） ──┐
               ├──→ 校对 → 监制 → 压制
时轴（不分段）──┘
```

**最简流程**（单人翻译+校对）：

```
翻译（不分段）→ 校对 → 压制
```

### 3.3 流水线管理命令

#### `/设置流水线 <番剧名> <流水线定义>`

用简洁的 DSL 语法定义工序链：

```
/设置流水线 淡岛百景 翻译[分段] → 时轴[分段] → 校对 → 后期 → 监制 → 压制
```

语法规则：
- 逗号分隔的工序**可并行**，共同作为下一步的前置依赖
- `→` 表示严格串行
- `[分段]` 标记该工序需要按时间段展开，不标记则默认不分段
- 可随时修改，修改后只影响新建的集，已有任务不受影响

#### `/查看流水线 <番剧名>`

查看当前配置的工序链。

#### 预设模板

首次绑定番剧时，如未设置流水线，使用默认模板：

```
翻译[分段] → 时轴[分段] → 校对 → 后期 → 监制 → 压制
```

### 3.4 依赖检查逻辑

当某人执行 `/完成` 时，Bot 根据该番剧的流水线配置执行依赖检查：

```
1. 更新当前任务进度 → "已完成"，写入完成时间
2. 查询同集、同工序的所有分段 → 是否全部完成？
   ├─ 否 → 仅回复"已标记完成，该工序还有 N 个分段未完成"
   └─ 是 → 继续步骤 3
3. 遍历流水线，找到所有 depends_on 包含当前工序的下游工序
4. 对每个下游工序，检查其 depends_on 中的所有工序是否都已全部完成
   ├─ 全部满足 → 群内通知"XX 现在可以接了！"
   └─ 部分满足 → 回复"XX 已全部完成，等待 YY 完成后 ZZ 即可开始"
```

### 3.5 特殊集类型

- **OP/ED 等特殊任务**：通过 `/新建特殊` 手动指定需要的工序类型（见 4.1 节），不受流水线模板约束
- **特别篇/OVA**：集数填 `OVA1`、`SP1` 等，使用番剧的标准流水线

## 四、命令体系

所有命令均有**群作用域**：

- **工作群**：Bot 收到命令时，通过 `group_index` 查询当前群绑定了哪些番剧，命令中的番剧名必须在绑定列表内，否则回复"该番剧未绑定到本群"
- **总群**：仅允许查询类命令（`/进度`、`/我的任务`、`/待接`、`/绑定列表`），可查看所有番剧的数据；写操作命令会被拒绝并提示"请到对应工作群操作"

### 4.1 管理员命令

#### `/绑定 <番剧名>`

将当前群与指定番剧绑定。Bot 自动查找腾讯文档中名为 `{番剧名}_工作表` 的子表并关联。

```
/绑定 淡岛百景
```

Bot 回复：`已将「淡岛百景」绑定到本群（123456789），对应子表：淡岛百景_工作表`

#### `/绑定id <fileID> <sheetID> [别名]`

通过文档 ID 和子表 ID 直接绑定，适用于子表命名不规范或跨文档的场景。

```
/绑定id abc123 sheet456 淡岛百景
```

Bot 回复：`已将子表 sheet456 绑定到本群，别名「淡岛百景」`

#### `/解绑 <番剧名>`

解除当前群与指定番剧的绑定。

#### `/绑定列表`

查看当前群绑定了哪些番剧，或全局查看所有绑定关系。

```
/绑定列表          ← 本群绑定的番剧（工作群）/ 所有番剧（总群）
/绑定列表 全部      ← 所有群的绑定关系（超级管理员，任意群可用）
```

#### `/设置流水线 <番剧名> <流水线定义>`

定义或修改番剧的工序链。详见 3.3 节。

```
/设置流水线 淡岛百景 翻译[分段] → 时轴[分段] → 校对 → 后期 → 监制 → 压制
/设置流水线 短片项目 翻译 → 校对 → 压制
```

#### `/查看流水线 <番剧名>`

查看当前配置的工序链。

#### `/新建集 <番剧名> <集数> [分段数=1]`（D12 简化）

创建一集的所有任务行并写入智能表。Bot 根据流水线配置自动展开：

```
/新建集 淡岛百景 7 3
```

假设流水线为 `翻译[分段] → 时轴[分段] → 校对 → 后期 → 监制 → 压制`，生成记录：

| 类型 | 集数 | 分段 | 组员 | 进度 | 完成时间 | 备注 |
|------|------|------|------|------|----------|------|
| 翻译 | 7 | 1 | | 未分配 | | |
| 翻译 | 7 | 2 | | 未分配 | | |
| 翻译 | 7 | 3 | | 未分配 | | |
| 时轴 | 7 | 1 | | 未分配 | | |
| 时轴 | 7 | 2 | | 未分配 | | |
| 时轴 | 7 | 3 | | 未分配 | | |
| 校对 | 7 | 0 | | 未分配 | | |
| 后期 | 7 | 0 | | 未分配 | | |
| 监制 | 7 | 0 | | 未分配 | | |
| 压制 | 7 | 0 | | 未分配 | | |

若流水线中不含某工序（如无后期），则不生成该工序的记录。标记为 `[分段]` 的工序按分段数展开（值 `1`..N），未标记的只生成一条「分段=0」记录。段范围信息（0-8 这类）不再自动写入，需要的话人工填到「备注」列。

执行后 Bot 在群内发布任务公告，列出所有可接任务。

#### `/新建特殊 <番剧名> <集数标识> <类型1> [类型2] [类型3] ...`

为 OP/ED 等特殊内容手动指定需要的工序类型，不受流水线模板约束，所有指定工序均不分段（分段=0）。

```
/新建特殊 淡岛百景 OP 翻译 时轴 校对 后期 压制
/新建特殊 淡岛百景 ED 翻译 校对 压制
```

生成记录（以第一条命令为例）：

| 类型 | 集数 | 分段 | 组员 | 进度 |
|------|------|------|------|------|
| 翻译 | OP | 0 | | 未分配 |
| 时轴 | OP | 0 | | 未分配 |
| 校对 | OP | 0 | | 未分配 |
| 后期 | OP | 0 | | 未分配 |
| 压制 | OP | 0 | | 未分配 |

特殊任务的依赖关系：仍然按流水线配置中相关工序的 `depends_on` 判断。如果指定的工序在流水线中不存在（如临时加的工序），则视为无依赖。

#### `/删除任务 <番剧名> <集数> [类型] [分段]`

删除指定任务记录。需要**二次确认**：Bot 先列出将被删除的记录摘要，用户回复"确认"后才执行删除。

```
/删除任务 淡岛百景 07
```

Bot 回复：

```
⚠️ 即将删除以下任务：
  淡岛百景 第07集，共 10 条记录
  其中 3 条已分配/进行中：翻译P1(@小明)、时轴P1(@阿杰)、时轴P2(@小王)
回复「确认删除」执行，30秒内有效
```

用户回复 `确认删除` → 执行删除。超时或回复其他内容 → 取消。

更精确的删除同样需要确认：

```
/删除任务 淡岛百景 07 时轴          ← 确认后删除第7集所有时轴任务
/删除任务 淡岛百景 07 时轴 P2       ← 确认后仅删除第7集时轴P2
```

#### `/修改任务 <番剧名> <集数> <类型> [分段] <字段>=<值>`

修改指定任务的某个字段。

```
/修改任务 淡岛百景 07 时轴 P2 备注=加急
/修改任务 淡岛百景 07 校对 进度=进行中
```

#### `/归档 <番剧名> <集数>`

将某一集所有"已完成"的任务批量标记为"归档"。

```
/归档 淡岛百景 07
```

### 4.2 组员命令

#### `/接活 <番剧名> <集数> <类型> [分段]`

认领一个任务。Bot 校验任务存在且进度为"未分配"后，写入组员 QQ 号并更新进度。

```
/接活 淡岛百景 07 时轴 P1
/接活 淡岛百景 07 校对            ← 不分段的工序无需指定分段
```

校验规则：
- 任务必须存在且进度为"未分配"
- 根据流水线配置检查前置工序是否全部完成
- 同一人不能同时持有超过 N 个未完成任务（可配置，防止一人揽太多）

#### `/完成 <番剧名> <集数> <类型> [分段]`

标记任务完成，自动写入当前日期作为完成时间，并触发依赖检查与下游通知。

```
/完成 淡岛百景 07 翻译 P1
```

校验规则：
- 只有任务的当前组员才能标记完成
- 进度必须为"已分配"或"进行中"

通知行为：
- 通知推送到该番剧绑定的**工作群**（不推送到总群）
- 组员标识为 QQ 号且在群内 → @该成员
- 组员标识为非 QQ 号（手动填表）→ 使用原始文本

#### `/放弃 <番剧名> <集数> <类型> [分段]`

放弃已接的任务，清空组员，进度回退为"未分配"，群内重新提醒该任务可接。

```
/放弃 淡岛百景 07 时轴 P1
```

#### `/进行中 <番剧名> <集数> <类型> [分段]`

将进度从"已分配"更新为"进行中"（可选，用于精细化进度追踪）。

### 4.3 查询命令（全部走本地缓存，不消耗 API）

查询命令在工作群和总群均可使用。在总群使用时可查看所有番剧的数据。

#### `/进度 <番剧名> [集数]`

展示某番或某集的任务状态一览。

```

```

Bot 回复示例：

```

#### `/我的任务`

#### `/待接 [番剧名]`

列出所有进度为"未分配"且前置工序已满足的可接任务。作用域同 `/我的任务`。

## 五、系统架构

### 5.1 整体数据流

```
QQ 群用户（总群 / 工作群 A / 工作群 B / ...）
   │
   │ 发送命令
   ▼
NapCat（QQ 协议层）
   │
   ▼
NoneBot2（命令路由）
   │
   ▼
nonebot-plugin-subflow
   ├── commands.py ──→ 解析命令参数 + 判断群类型（总群/工作群）+ 校验权限
   ├── task_manager.py ──→ 业务逻辑（校验、依赖检查、流水线解析）
   ├── cache.py ──→ 本地内存缓存（查询走这里，含 group_index）
   ├── storage/
   │   └── tencent_doc.py ──→ 腾讯文档 API（写操作走这里）
   └── messenger/
       └── qq.py ──→ QQ 消息发送（@成员 / 文本回退）
              │                         │
              ▼                         ▼
        腾讯文档智能表              通知推送到工作群（不推送总群）
```

# ──── NoneBot2 基础 ────

#### 自动生成策略

| 文件 | 首次启动行为 | 说明 |
|------|-------------|------|
| `.env` | **不自动生成**，缺失则启动报错并提示从 `.env.example` 复制 | 含敏感信息，必须手动配置 |
| `data/bindings.json` | 自动创建空结构 `{"main_group_id": null, "bindings": {}}` | 运行时通过 `/绑定` 命令填充 |
| `data/pipelines.json` | 自动创建空结构 `{}` | 运行时通过 `/设置流水线` 命令填充，未配置的番剧使用 `SUBFLOW_DEFAULT_PIPELINE` |

`config.py` 使用 NoneBot2 标准的 Pydantic BaseSettings，从 `.env` 读取环境变量，类型校验失败时在启动阶段即报错。

### 5.5 缓存策略详细设计

```
┌─────────────────────────────────────────────────┐

│                                                  │

│    "淡岛百景": {                                  │

│      "pipeline": [...],                          │

│        "record_id_1": {                          │

│          "完成时间": "2026-05-20",                │

│        },                                        │
│        ...                                       │
│      }                                           │
│    },                                            │
│    ...                                           │
│  }                                               │
│                                                  │
│  # 反向索引：群号 → 番剧名列表，用于快速判断       │
│  # 当前群可操作哪些番剧                           │
│  group_index = {                                 │
│    123456789: ["淡岛百景", "葬送的芙莉莲"],        │
│    223456789: ["孤独摇滚"],                       │
│  }                                               │
│                                                  │

│                                                  │
│  last_sync_time: "2026-05-25T14:30:00"           │
└─────────────────────────────────────────────────┘

同步策略：
  ✦ 冷启动：全量拉取所有子表及记录 → 约消耗 N 次 API（N = 子表数 × 2）
  ✦ 定时同步：每 30 分钟全量刷新一次 → 每天约 96 次 API
  ✦ 写穿透：每次写操作 API 成功后，立即同步更新本地缓存
  ✦ 查询零消耗：所有 /进度、/我的任务、/待接 命令直接读缓存
```

```

## 七、后续扩展方向

- **催活提醒**：任务分配超过 N 天未完成，Bot 自动 @ 提醒组员
- **统计报表**：`/统计 <番剧名>` 查看各成员完成任务数、平均耗时
- **权限细分**：某些人只能接特定工序（如只能接翻译）
- **飞书迁移**：实现 `storage/feishu.py` + `messenger/feishu.py` 即可切换至飞书，业务逻辑不变
- **AstrBot 迁移**：重写 `commands.py` 适配 AstrBot 的命令系统，其余模块不变
